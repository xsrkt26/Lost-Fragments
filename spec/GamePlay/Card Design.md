## 二、 技术实现方案 (Technical Design)

### 1. 核心系统架构重构
为了支持复杂的卡牌效果，系统将从硬编码转向**数据驱动 (Data-Driven)**：
*   **ItemData 资源化**：每种卡牌将定义为 `res://data/items/` 下的 `.tres` 文件。
*   **ItemDatabase (Autoload)**：全局物品库，负责启动时加载所有卡牌资源并提供随机抽取接口。
*   **多触发器支持**：扩展 `ItemEffect` 以支持 `OnHit` (被撞), `OnDraw` (抽到), `OnDiscard` (丢弃), `OnGlobal` (全局监听)。

### 2. 技术映射与组件需求
根据卡牌功能，需开发以下核心技术组件：

| 机制分类     | 涉及卡牌       | 技术组件/需求                                            |
| :------- | :--------- | :------------------------------------------------- |
| **条件判定** | 破旧闹钟, 英语课本 | `ConditionalEffect`: 增加 `check_condition()` 逻辑。    |
| **路径修改** | 锈齿轮, 数学课本  | `ImpactModifier`: 允许修改 `ImpactResolver` 的传播方向或过滤器。 |
| **状态转换** | 苹果, 礼物盒    | `TransformationEffect`: 实现物品在网格中的原位替换。             |
| **全局监听** | 垃圾回收器, 燃料罐 | `GlobalEventBus`: 建立全局事件总线，监听撞击与触发。                |
| **生成逻辑** | 纸团, 苹果核    | `SpawnSystem`: 在背包寻找空位并实例化新物品。                     |

## 三、 开发路线图 (Roadmap)

### 第一阶段：数据与动态加载 (当前目标)
1.  **扩展 ItemData**: 增加 `price`, `base_cost`, `category_tags` 等字段。
2.  **建立数据库**: 实现 `ItemDatabase.gd`，支持自动扫描资源文件夹。
3.  **重构抽卡**: `BattleManager` 调用 `ItemDatabase.get_random_item()`。

### 第二阶段：效果系统扩展
1.  实现 `OnDiscard` 钩子，打通垃圾桶逻辑。
2.  实现基础的 `ConditionalEffect` (如基于抽卡次数的判定)。

### 第三阶段：高级机制开发
1.  开发 `ImpactResolver` 的多向传导逻辑（支持齿轮）。
2.  开发物品转换与生成系统。
