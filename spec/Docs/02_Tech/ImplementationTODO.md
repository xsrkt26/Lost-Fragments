# 系统实现 TODO

本文档记录当前代码与最新策划案之间的系统级差距。范围不包含道具系统；道具系统单独处理。

## 优先级说明

- P0：主流程骨架，其他系统依赖它接入。
- P1：核心局内规则，会直接影响战斗胜负与长期构筑。
- P2：成长、奖励与经济系统，影响完整一局游戏体验。
- P3：表现、调试与体验完善，不阻塞规则闭环。

## P0：先完成主流程骨架

### 1. 局外路线与节点推进

当前状态：整备室中可以自由进入战斗、商店和图鉴。

实现状态：已完成基础版本。

本次实现范围：

- 新增配置化路线定义，默认路线暂为：局内游戏 -> 商店 -> 事件 -> 局内游戏 -> 商店 -> 事件 -> Boss局内游戏 -> 商店 -> 事件。
- 路线配置集中在 `RouteConfig`，后续可以扩展 `cutscene`、`reward`、`elite_battle` 等节点类型。
- `RunManager` 保存当前路线 ID、当前场景层、当前节点下标和已完成节点。
- Hub 改为路线按钮驱动，只允许进入当前节点，已完成和未解锁节点不可进入。
- 进入节点前会驱动角色移动到对应节点位置。
- 战斗胜利、商店离开、事件确认后会推进到下一个路线节点。
- 事件节点已接入占位事件场景，用于验证路线闭环。
- Boss 战节点已作为节点类型保留，当前暂时进入普通战斗场景。

遗留到后续优先级：

- Boss 分数目标和失败规则在 P0-2 / P0-3 中实现。
- 6 个场景结束后的整局胜利在 P0-3 中实现。
- 路线配置后续可从脚本常量迁移到 `.tres`、JSON 或策划表。

策划目标：

- 玩家开始新游戏后进入局外路线界面。
- 节点按固定顺序推进：局内游戏 -> 商店 -> 事件 -> 局内游戏 -> 商店 -> 事件 -> Boss局内游戏 -> 商店 -> 事件。
- 后续节点可以展示，但当前未解锁节点不可点击。
- 玩家点击当前节点后，角色移动到节点并进入对应场景。
- 战斗、商店、事件结束后回到路线界面，并推进到下一个节点。

实现建议：

- 在 `RunManager` 中新增当前场景、当前节点索引、节点类型、Boss 标记。
- 将 Hub 从自由入口改为路线节点控制器。
- 节点完成后统一通过 `RunManager.advance_node()` 推进。

### 2. 战斗类型与分数目标规则

当前状态：每一局都有目标分数，达到目标会立即胜利。

策划目标：

- 普通局内游戏不一定有分数目标，没有目标时显示“无”。
- Boss 局内游戏必须有分数目标。
- 分数目标应在战斗结束时检查，而不是达到目标后立即结束。
- 未完成 Boss 分数要求时整局失败。

实现建议：

- 新增 `BattleConfig` 或在 `RunManager` 中提供当前战斗配置。
- UI 支持目标分数为“无”。
- 移除“达到目标自动胜利”的局内逻辑，改为结束时统一判定。

### 3. 六场景与 Boss 通关结构

当前状态：只有 `current_depth` 递增，没有场景、Boss 和整局胜利。

策划目标：

- 游戏由 6 个场景构成。
- 每个场景包含固定节点序列，并以 Boss 局内游戏作为关键检查点。
- 通过 Boss 后进入下一个场景。
- 完成第 6 个场景后整局胜利，可以开始新游戏。

实现建议：

- 将 `current_depth` 拆分或替换为 `current_act`、`current_node_index`、`current_battle_index`。
- Boss 胜利后判断是否进入下一场景或结束整局。

## P1：补齐局内核心规则

### 4. 战斗生命周期

当前状态：梦值归零直接触发失败弹窗；战斗达到目标会自动结束。

策划目标：

- 捕梦后先完整结算抽到物品和背包内物品效果。
- 梦值归零后，在本次结算完成后自动结束战斗。
- 玩家也可以手动结束战斗。
- 战斗结束时统一处理胜负、奖励和战后清理。

实现建议：

- 给 `BattleManager` 增加明确的战斗状态机：抽取中、结算中、可操作、结束中。
- 将 `GameState.game_over` 从“立即失败”改为“请求结束战斗”。
- 战斗结束统一走一个 `finish_battle()` 入口。

### 5. 背包跨战斗状态

当前状态：每场战斗从卡组洗牌，背包布局不跨战斗保存。

策划目标：

- 进入局内时展示玩家当前背包物品布局。
- 背包内物品的摆放、旋转和保留状态需要在战斗之间延续。
- 背包外物品在战斗结束时自动丢弃。
- 污染在每场战斗结束后清除。
- 衍生物品在每场战斗结束后从背包中消失。

实现建议：

- 在 `RunManager` 中保存背包物品实例列表，包括物品 id、位置、方向、形状、运行时状态。
- 战斗开始时从运行数据重建背包。
- 战斗结束时写回背包状态，并执行污染清理与衍生物品移除。

### 6. 播种与梦境之种成长

当前状态：已有梦境之种物品资源，但缺少统一的播种和成长机制。

策划目标：

- 播种时，如果指定方向第一个格子为空，生成 1 级梦境之种。
- 如果目标格已有梦境之种，则梦境之种等级 +1。
- 梦境之种升级需要触发相关物品和饰品效果。
- 梦境之种属于衍生物品时，应遵循战后清理规则。

实现建议：

- 新增统一的 `sow_seed()` / `upgrade_seed()` 逻辑入口。
- 通过事件总线广播播种、升级、收获等事件，供物品和饰品监听。

### 7. 多源撞击调度

当前状态：已有单条撞击链解析，但缺少多个物品同时发起撞击时的全局调度规则。

策划目标：

- 多个物品同时发起撞击时，按左上优先级排序。
- 每条撞击链完整结算后，再处理下一条撞击链。
- 同一结算窗口内的新增撞击需要进入队列，避免顺序不确定。

实现建议：

- 新增撞击队列，由 `BattleManager` 或专门的 `ImpactQueue` 管理。
- 队列项记录来源物品、方向、优先级和触发原因。
- 所有延迟撞击统一入队，不直接 `call_deferred("trigger_impact_at")`。

## P2：补齐长期构筑与经济

### 8. 饰品系统

当前状态：只有空的 `current_ornaments` 状态和局内 UI 饰品栏，没有饰品数据和效果。

策划目标：

- 实现 50 个饰品的数据表和资源加载。
- 饰品不占背包格、不参与撞击、不能重复获得。
- 饰品通过商店、事件、关卡奖励或 Boss 奖励获得。
- 鼠标悬停显示名称、稀有度、最早出现层、标签、效果和说明。
- 多个饰品同时触发时按获得顺序结算。
- 数值修改类效果先结算“减少消耗”和“改变规则”，再结算“获得分数”和“获得梦值”。

实现建议：

- 新增 `OrnamentData`、`OrnamentDatabase`、`OrnamentEffect`。
- 将饰品触发点接入捕梦、撞击链、污染、净化、播种、丢弃、战斗结束等事件。
- 先实现通用饰品和少量污染/机械饰品，再扩展完整 50 个。

### 9. 关卡奖励与 Boss 奖励

当前状态：战斗胜利只给固定碎片奖励。

策划目标：

- 普通关卡和 Boss 结束后可以提供奖励。
- 奖励可以包含物品、饰品、碎片或其他长期资源。
- Boss 奖励应优先提供更高价值的构筑方向。

实现建议：

- 新增奖励池和奖励选择 UI。
- 奖励生成基于当前场景层数、节点类型、稀有度和已有构筑。

### 10. 商店系统

当前状态：商店随机出售 3 个物品，购买后直接加入卡组。

策划目标：

- 商店作为局外路线节点出现。
- 商店应支持物品和饰品出售。
- 商品池需要按层数、稀有度、价格和已拥有内容过滤。
- 购买后更新长期构筑状态，并推进或允许返回路线。

实现建议：

- 将商店商品生成从简单随机改为基于奖励池。
- 接入饰品商品、价格、库存和刷新规则。
- 购买物品时需要明确加入卡组或背包的规则。

### 11. 事件系统

当前状态：没有事件场景和事件池。

策划目标：

- 事件是局外路线节点之一。
- 事件可以提供物品、饰品、碎片、梦值、背包变化或风险选择。
- 事件结束后返回路线并推进到下一个节点。

实现建议：

- 新增 `EventData`、`EventDatabase` 和事件选择 UI。
- 先实现少量基础事件，用于验证路线推进和奖励系统。

## P3：体验与表现完善

### 12. 捕梦表现与抽取区域

当前状态：捕梦按钮可以抽物品，物品出现在捕梦区域附近，但动画和流程表现较简单。

策划目标：

- 点击捕梦按钮后播放捕梦动画。

实现建议：

- 给捕梦流程加入动画回调和输入锁。
- 将抽到物品的出生点配置化。

### 13. UI 信息完整性

策划目标：

- 分数区域支持目标为“无”。
- 饰品栏展示已获得饰品并支持悬停说明。
- 背包锁定格、可用格、丢弃区和抽取区需要与最终 UI 规范一致。

实现建议：
