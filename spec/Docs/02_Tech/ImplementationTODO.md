# 系统实现 TODO

本文档记录当前代码与最新策划案之间的系统级差距。范围不包含道具系统；道具系统单独处理。

## 优先级说明

- P0：主流程骨架，其他系统依赖它接入。
- P1：核心局内规则，会直接影响战斗胜负与长期构筑。
- P2：成长、奖励与经济系统，影响完整一局游戏体验。
- P3：表现、调试与体验完善，不阻塞规则闭环。

## P0：先完成主流程骨架

### 3. 六场景与 Boss 通关结构

当前状态：只有 `current_depth` 递增，没有场景、Boss 和整局胜利。

实现状态：已完成基础版本。

本次实现范围：

- 路线推进到第 6 场景最后一个节点后，自动触发整局胜利，关闭当前 run，并发出 `run_finished(true)`。

- 通过 Boss 后进入下一个场景。
- 完成第 6 个场景后整局胜利，可以开始新游戏。

实现建议：

- 将 `current_depth` 拆分或替换为 `current_act`、`current_node_index`、`current_battle_index`。
- Boss 胜利后判断是否进入下一场景或结束整局。

策划目标：

- 捕梦后先完整结算抽到物品和背包内物品效果。
- 梦值归零后，在本次结算完成后自动结束战斗。

- 将 `GameState.game_over` 从“立即失败”改为“请求结束战斗”。
- 战斗结束统一走一个 `finish_battle()` 入口。

### 5. 背包跨战斗状态

当前状态：每场战斗从卡组洗牌，背包布局不跨战斗保存。

实现状态：已完成基础版本。

本次实现范围：

- `RunManager` 新增 `current_backpack_items`，保存跨战斗背包布局。
- 战斗结束弹窗前，先丢弃背包外物品，再保存背包内物品状态。
- 保存字段包含：物品 id、根坐标、方向、形状和 runtime id。
- 污染不保存，下一场战斗恢复时从 0 层开始。
- 带有 `衍生物品` 标签的物品不保存，下一场战斗不会恢复。
- 新战斗初始化时，`BattleManager` 从 `RunManager.current_backpack_items` 重建背包逻辑实例。
- `MainGameUI` 会为已恢复的背包物品生成对应 `ItemUI`，保持可拖拽、可旋转、可丢弃。
- 存档序列化新增 `backpack_items` 字段，旧存档默认空背包，保持兼容。
- 物品变身、进化或替换数据时会保留原 `runtime_id`，并通知战斗 UI 同步物品显示与映射，避免苹果核变回苹果等效果后丢失拖拽、悬浮或动画定位。

自动化测试：

- `test/unit/test_backpack_persistence.gd` 覆盖背包状态保存、衍生物品过滤、污染不跨战斗保存，以及恢复时保留位置、方向、形状和 runtime id。
- `test/unit/test_backpack_manager.gd` 覆盖替换同形状和不同形状物品时保留 `runtime_id`，并发出替换事件。

实现假设：

- 当前只持久化背包内物品；背包外未放置物品仍按策划在战斗结束时自动丢弃。
- “保留状态”字段尚未有明确数据模型，本次未新增独立保留标记；后续如果策划需要锁定/保留类效果，可在 `current_backpack_items` 中追加字段。
- 当前通过 `衍生物品` 标签判断战后移除，例如 `apple_core`。

策划目标：

- 进入局内时展示玩家当前背包物品布局。
- 背包内物品的摆放、旋转和保留状态需要在战斗之间延续。
- 背包外物品在战斗结束时自动丢弃。
- 污染在每场战斗结束后清除。
- 衍生物品在每场战斗结束后从背包中消失。

实现建议：

- 在 `RunManager` 中保存背包物品实例列表，包括物品 id、位置、方向、形状、运行时状态。
- 战斗开始时从运行数据重建背包。
- 战斗结束时写回背包状态，并执行污染清理与衍生物品移除。

### 6. 播种与梦境之种成长

当前状态：已有梦境之种物品资源，但缺少统一的播种和成长机制。

实现状态：已完成基础版本。

本次实现范围：

- `BackpackManager` 新增统一入口 `sow_seed(source_instance, direction, item_db, levels = 1)`。
- 如果来源物品指定方向第一格为空，生成 `dream_seed_1x1`。
- 通过播种生成的梦境之种会追加 `衍生物品` 标签，战斗结束后不会跨战斗保存。
- 如果指定方向第一格已有梦境之种，则调用 `upgrade_seed()` 进行等级提升。
- 梦境之种最高提升到 5 级，对应 `dream_seed_1x1` 到 `dream_seed_5x5`。
- 升级时保留 runtime id 和方向；如果升级后形状放不下，会回滚到升级前状态。
- `GlobalEventBus` 新增 `seed_sown(instance)` 与 `seed_upgraded(instance, old_level, new_level)`，供后续饰品、道具和奖励监听。

实现假设：

- “所指方向第一格”按来源物品形状边缘外的第一格计算；多格物品同方向有多个候选格时使用左上优先。
- 当前只是底层统一入口，尚未把具体物品或道具效果改成播种效果；道具系统和饰品系统接入时复用该入口。

策划目标：

- 播种时，如果指定方向第一个格子为空，生成 1 级梦境之种。
- 如果目标格已有梦境之种，则梦境之种等级 +1。
- 梦境之种升级需要触发相关物品和饰品效果。
- 梦境之种属于衍生物品时，应遵循战后清理规则。

实现建议：

- 新增统一的 `sow_seed()` / `upgrade_seed()` 逻辑入口。
- 通过事件总线广播播种、升级、收获等事件，供物品和饰品监听。

### 7. 多源撞击调度

当前状态：已有单条撞击链解析，但缺少多个物品同时发起撞击时的全局调度规则。

实现状态：已完成基础版本。

本次实现范围：

- `BattleManager` 新增统一撞击队列 `queue_impact_at(pos, direction, source, reason)`，旧入口 `trigger_impact_at(pos)` 保留并改为入队。
- 队列项记录来源物品、方向、触发原因、左上优先级和入队序号；优先级按 `y * grid_width + x` 计算，保证同一行从左到右、行与行从上到下。
- `request_draw()` 在抽取物品和全局抽取监听结束后，会等待本结算窗口内的撞击队列处理完成，再回到可操作状态或发出 pending 结束请求。
- 每次只弹出一条撞击链，等待 `ImpactResolver` 与 `SequencePlayer` 完整结算后再处理下一条；结算过程中新增的撞击继续进入同一队列并重新排序。
- `interval_trigger`、`reactive_impact`、`same_item_draw`、`stain_magnifier`、`tag_reactive` 等延迟撞击效果已统一改为入队，不再直接 `call_deferred("trigger_impact_at")`。

实现假设：

- “同时发起”以同一抽取/撞击结算窗口内进入队列的撞击为准；新入队的撞击会参与剩余队列排序，但不会打断已经开始播放的当前撞击链。
- 如果来源物品已离开背包或不再有效，该队列项会被跳过，避免已删除物品触发幽灵撞击。

自动化测试：

- `test/unit/test_impact_queue.gd` 覆盖左上优先级排序、同窗口新增撞击重新排序、空格/失效来源拒绝，以及旧 `trigger_impact_at()` 入口兼容入队。

策划目标：

- 多个物品同时发起撞击时，按左上优先级排序。
- 每条撞击链完整结算后，再处理下一条撞击链。
- 同一结算窗口内的新增撞击需要进入队列，避免顺序不确定。

实现建议：

- 新增撞击队列，由 `BattleManager` 或专门的 `ImpactQueue` 管理。
- 队列项记录来源物品、方向、优先级和触发原因。
- 所有延迟撞击统一入队，不直接 `call_deferred("trigger_impact_at")`。

### 8. 饰品系统

当前状态：只有空的 `current_ornaments` 状态和局内 UI 饰品栏，没有饰品数据和效果。

- `万向轴承`、`反冲片` 等会改变撞击传导结构的饰品，当前通过战斗管理器的确定性撞击队列追加额外撞击表达，不在 `ImpactResolver` 中途改写本条链的扫描规则。
- `回收券` 的折扣在购买结算时生效，商店 UI 会在成功购买后刷新未购买商品显示价格。
- `空梦奖杯` 当前按奖励选项总数 +1 实现；后续如果需要“只增加饰品选项而不增加总选项”，需要扩展 `RewardGenerator` 的饰品槽位控制。

- 饰品通过商店、事件、关卡奖励或 Boss 奖励获得。

### 9. 关卡奖励与 Boss 奖励

当前状态：战斗胜利只给固定碎片奖励。

实现状态：已完成基础版本。

本次实现范围：

- 新增 `RewardGenerator`，统一生成奖励选项，奖励类型包含物品、饰品和碎片。
- 普通战斗胜利默认生成 3 个选项：可加入卡组的物品、当前层可获得且未拥有的饰品、碎片。
- Boss 战奖励使用同一入口，但优先选择更高稀有度和更高价格的饰品；第 4 层及以后会优先提供稀有饰品。
- `RunManager` 新增 `generate_current_reward_options()` 与 `apply_reward()`，奖励写入长期状态：碎片增加、物品加入卡组、饰品按获得顺序加入且不能重复。
- 胜利弹窗改为显示奖励选项，玩家选择奖励后再调用 `win_battle(0)` 推进路线；固定碎片奖励不再直接发放。

实现假设：

- 当前奖励生成是确定性基础版本，用于验证奖励管线；后续可以在 `RewardGenerator` 内替换为带权重随机、按流派倾向、按已有构筑过滤的奖励池。
- 物品奖励当前直接加入卡组；如果后续策划要求“加入背包”或“暂存待摆放”，在 `RunManager.apply_reward()` 的物品分支扩展。
- 选择奖励是必选流程；无可用奖励时才显示普通继续按钮。

自动化测试：

- `test/unit/test_reward_system.gd` 覆盖普通战斗奖励类型、Boss 稀有饰品优先级，以及碎片、物品、饰品奖励写入长期状态和重复饰品拒绝。

策划目标：

- 普通关卡和 Boss 结束后可以提供奖励。
- 奖励可以包含物品、饰品、碎片或其他长期资源。
- Boss 奖励应优先提供更高价值的构筑方向。

实现建议：

- 新增奖励池和奖励选择 UI。
- 奖励生成基于当前场景层数、节点类型、稀有度和已有构筑。

### 10. 商店系统

当前状态：商店随机出售 3 个物品，购买后直接加入卡组。

实现状态：已完成基础版本。

本次实现范围：

- 新增 `ShopGenerator`，统一生成商店商品，商品类型包含物品和饰品。
- 商店物品池从 `ItemDatabase.get_all_items()` 获取，只出售可抽取物品，并按价格与 id 稳定排序。
- 商店饰品池从 `OrnamentDatabase.get_available_ornaments()` 获取，按当前场景层数和已拥有饰品过滤，同名饰品不会出现在货架中。
- `RunManager` 新增 `generate_current_shop_offers()` 与 `buy_shop_offer()`，购买会统一扣除碎片并写入长期状态：物品加入卡组，饰品加入 `current_ornaments`。
- 商店 UI 改为展示物品/饰品类型、稀有度和价格，悬停显示描述；购买成功后对应按钮禁用为“已购买”。
- 离开商店仍沿用路线节点推进逻辑，允许玩家买完后主动返回路线。

实现假设：

- 当前商店生成是确定性基础版本，优先验证商品池与购买管线；后续可在 `ShopGenerator` 中替换为带权重随机、库存刷新和按构筑倾向过滤。
- 物品购买当前直接加入卡组；如果后续策划要求购买后进入背包或暂存区，需要扩展 `buy_shop_offer()` 的物品分支。
- 饰品使用策划表推荐价格；前期可能出现买不起的饰品，这是经济数值调试项，当前不在本次逻辑实现范围内。

自动化测试：

- `test/unit/test_shop_system.gd` 覆盖商店物品/饰品商品生成、已拥有饰品过滤、购买扣碎片、物品入卡组、饰品入长期状态和重复饰品购买拒绝。

策划目标：

- 商店作为局外路线节点出现。
- 商店应支持物品和饰品出售。
- 商品池需要按层数、稀有度、价格和已拥有内容过滤。
- 购买后更新长期构筑状态，并推进或允许返回路线。

实现建议：

- 将商店商品生成从简单随机改为基于奖励池。
- 接入饰品商品、价格、库存和刷新规则。
- 购买物品时需要明确加入卡组或背包的规则。

### 11. 事件系统

当前状态：已完成基础版本。

本次实现范围：

- 新增 `data/events/events.json`，首批事件包含碎片、物品、饰品、梦值回复、背包空间扩展和带碎片成本的风险选择。
- 新增 `EventData` 与 `EventDatabase`，作为 autoload 统一加载事件池，并按当前场景层过滤可用事件。
- `RunManager` 新增 `apply_event_choice()`，事件效果统一写入长期状态：碎片、物品、饰品、背包可用区域和梦值。
- `RunManager` 新增背包可用区域长期字段，默认 7x7 总格、5x5 可用，事件最多扩展到 7x7；`BattleManager` 初始化时读取该配置。
- 事件场景从占位确认改为选择式 UI，玩家选择事件选项后推进路线并回到 Hub；有待选择项时不会通过 ESC 跳过事件。

实现假设：

- 事件池当前使用确定性选择，按当前层数和路线节点位置从可用事件中取一项，用于保证测试和调试稳定；后续可以替换为带权重随机和事件去重。
- 事件获得物品当前沿用奖励系统语义，直接加入卡组；如果后续策划要求进入背包或暂存区，需要扩展事件效果类型。
- 梦值回复通过当前 `GameState` 立即生效，主要用于局外事件进入下一场战斗前的状态调整；若后续需要跨场景梦值上限或长期梦值资源，应拆出独立运行状态字段。
- 背包变化当前只实现“可用区域扩展”，未实现删格、锁格、强制移动已有物品等高风险变化。

自动化测试：

- `test/unit/test_event_system.gd` 覆盖事件池加载与层数过滤、确定性事件选择、事件选项写入长期状态、成本不足/重复饰品时回滚，以及背包空间扩展的存档和战斗初始化读取。

策划目标：

- 事件是局外路线节点之一。
- 事件可以提供物品、饰品、碎片、梦值、背包变化或风险选择。
- 事件结束后返回路线并推进到下一个节点。

实现建议：

- 新增 `EventData`、`EventDatabase` 和事件选择 UI。
- 先实现少量基础事件，用于验证路线推进和奖励系统。
