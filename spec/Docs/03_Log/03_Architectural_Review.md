# 当前架构评审记录

评审日期：2026-05-14

文档状态：已按当前代码更新。本文是评审记录，不作为需求来源。

## 1. 当前架构判断

项目已经从早期单局背包撞击原型，推进到可运行的 6 层路线闭环：

- Hub 路线节点推进。
- 普通/Boss 战斗规则。
- 背包跨战斗持久化。
- 播种、梦境之种和多源撞击队列。
- 饰品、奖励、商店、事件、经济配置。
- 全量 GUT、严格场景冒烟和本地发布预检。

整体架构仍以数据驱动和逻辑/表现分离为主。`RunManager` 是整局状态源，`BattleManager` 是局内控制器，`BackpackManager`/`ImpactResolver`/`SequencePlayer` 分别负责网格、解析和表现播放。

## 2. 已解决的早期风险

- 资源共享污染：通过 `ItemInstance`、深拷贝和 runtime id 保留解决。
- 表现与结算脱节：通过 `GameAction` 和 `SequencePlayer` 缓解。
- 多源撞击顺序不确定：通过 `BattleManager` 撞击队列统一调度。
- 奖励/商店/事件随机不可复现：通过 `RunManager` 可序列化随机源解决。
- Autoload/class_name 解析风险：严格场景冒烟会先执行 headless editor 导入，再加载关键场景。
- 发布流程不可复现：已新增本地导出脚本和 manifest。

## 3. 当前主要风险

### 3.1 UI 复用边界

`main_game_ui.tscn` 同时承担局内战斗和 Hub 整理背包浮层。当前已增加整理背包浮层模式，避免进入 BATTLE 输入上下文、重置局内状态或播放战斗 BGM，并提供鼠标关闭按钮和居中背包布局。

建议：基础体验问题已修复；如果后续整理背包加入更多专属功能，再拆出独立整理背包 scene，避免 `main_game_ui` 继续膨胀。

### 3.2 复杂机械饰品仍需精修

正式机械物品已接入左/右传动、双向传动、全向传动和撞击结算上下文统计。齿轮油、万向轴承、反冲片已完成 v1.2 口径修正；后续风险主要是如果策划继续增加更复杂的机械饰品，可能需要更细的动作元数据。

建议：暂不做额外底层重构，等新饰品或新道具确实需要时再扩展 `GameAction` 元数据。

### 3.3 道具系统尚未进入架构

F1 已由用户确认暂缓。后续恢复时必须独立于普通背包格实现，不应把道具硬塞进 `ItemData` 或 `BackpackManager` 的普通物品流程。

51-56 号道具联动饰品已经以未启用数据进入总表，并通过奖励/商店可用池过滤和 no-op 效果保护运行时。恢复 F1 时只需要补独立道具事件与对应饰品效果，再打开启用状态。

### 3.4 视觉资产仍是占位

正式捕梦动画、美术或 CG 缺少素材。当前只实现轻量 tween 与通用按钮反馈。

## 4. 建议优先级

1. 若恢复道具需求，先建立独立道具数据和栏位模型。
2. 基于实机游玩数据继续调经济曲线。
3. 确认发布权限后接 GitHub Releases。

## 5. 评审结论

当前架构足以支撑继续迭代主流程和局内规则。下一阶段最值得投入的是道具系统恢复条件确认、实机体验数据和发布自动化，而不是重新做大规模底层重构。
